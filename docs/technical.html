Technical Reference

Architecture rationale, HTTP API, WebSocket protocol, hardware details, and theme internals.

Architecture & Design Rationale

StepStick is designed to be a local-first, low-latency device. Several key engineering decisions were made to optimize for the limited 250mAh battery of the M5StickS3 and to streamline the user experience for non-technical streamers.

Improv Serial vs. Captive Portal

Traditional IoT setups rely on captive portals (e.g., WiFiManager), which require the ESP32 to run a temporary AP, a DNS server, and an HTTP server. This consumes significant heap memory, delays the boot sequence, and causes friction on mobile devices. By relying exclusively on the Improv Wi-Fi serial standard, we offload provisioning to the user's browser via Web Serial. This reduces the firmware footprint and allows for a seamless, one-click setup directly following the firmware flash.

Event-Driven WebSockets over HTTP Polling

If OBS were configured to poll the device for step updates via standard HTTP requests, the ESP32 radio would remain constantly active, severely degrading battery life. Instead, StepStick acts as a WebSocket server. OBS connects once, and the ESP32 only broadcasts a payload when the BMI270 IMU registers an explicit step increment or a state transition. During periods of inactivity, network traffic is zero.

Dynamic CSS Injection

To allow users to theme their overlay without touching OBS settings, theme preferences are stored in ESP32 Non-Volatile Storage (NVS). The /theme.css endpoint dynamically generates a stylesheet based on these NVS values. When a user updates the theme via the web UI, the device broadcasts a {"command": "reload"} WebSocket payload. OBS receives this, automatically refreshes the browser source, and pulls the newly generated CSS, creating a real-time remote-control experience.

HTTP Endpoints

All endpoints are served on port 80.

Method Path Description
GET / Configuration UI with theme editor and OBS URL helper
GET /overlay Overlay HTML — use this as the OBS Browser Source URL
GET /theme.css Dynamically generated CSS with current theme variables; fetched by the overlay on load
POST /savetheme Persist theme settings. Broadcasts a reload command to all WebSocket clients, then redirects to /?saved=1
POST /reset Reset the displayed step count. Redirects to /

WebSocket API

The device runs a WebSocket server on port 81. The overlay connects to ws://<device-ip>:81/ and reconnects automatically on disconnect.

Server → Client messages

Sent immediately on connect, and again whenever step count or activity state changes.

{
  "steps": 1234,       // current displayed step count (integer)
  "state": 1,          // activity state: 0=STILL, 1=WALKING, 2=RUNNING
  "activity": "🚶",    // icon string — emoji or image URL depending on theme
  "battery": 82        // battery percentage (0–100), polled every 30 s
}

Also sent when theme is saved:

{ "command": "reload" }   // instructs overlay to reload the page
💡 The overlay only broadcasts when steps or state actually changes, so there is no polling traffic during idle periods.

Activity States

Value Name Default icon CSS class on overlay
0 STILL 🧍 (none — opacity 0.3)
1 WALKING 🚶 active-walk
2 RUNNING 🏃 active-run

Activity values come directly from the BMI270 step-activity feature. The sensor only transitions from STILL when motion begins; it never re-reports STILL once motion has started during a session (the firmware preserves the last active state).

Theme System

Theme values are stored in NVS and served as CSS custom properties via /theme.css. The overlay loads this stylesheet on every page load.

CSS custom properties

Property Default Description
--bg-color #0f0f14e6 HUD background (background colour + e6 alpha suffix for 90 % opacity)
--text-color #ffffff Step count and label text
